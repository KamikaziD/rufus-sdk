Rufus SDK — Architecture

How it's built, from the inside out.

01 What is Rufus?

Rufus is a workflow engine for fintech edge devices — POS terminals, ATMs, mobile readers, browsers, and cloud servers — all running the same code. You define a workflow once in YAML, and the engine runs it wherever you deploy it. The same payment-processing logic that runs on an offline ATM also runs in a browser and in a cloud API server, without modification.

02 The Core Engine

A workflow is a YAML file plus a set of Python functions. The engine (the Workflow class in src/rufus/workflow.py) reads the YAML, calls the functions in order, and tracks state in a typed Python object (a Pydantic BaseModel subclass). That's it.

Each step function gets the current state and returns a dict of changes. The engine merges those changes in, persists the new state, and moves to the next step. 

# A minimal step function
def validate_card(state, context, **user_input) -> dict:
    if state.amount > state.floor_limit:
        return {"requires_online_auth": True}
    return {"requires_online_auth": False}

The engine itself has no opinion on databases, task queues, or deployment targets. Those are supplied by the six provider interfaces described next.

Hot-path DTOs use msgspec.Struct. Persistence providers return typed structs (WorkflowRecord, TaskRecord, MetricRecord) instead of plain dicts. msgspec.json.decode(bytes, type=WorkflowRecord) constructs the struct directly from bytes with no intermediate dict allocation — the typed-decode fast path gives a 5–10× throughput improvement on the DTO round-trip vs a dict decode.

03 The Six Plugs (Provider Pattern)

The Workflow constructor requires exactly six provider arguments. Pass None for any of them and you get a ValueError at startup — not a silent failure at runtime. Each is an abstract base class; you can swap any implementation without touching the engine.

PersistenceProvider
Storage
Reads and writes workflow state, audit logs, and task records. 16 required abstract methods; 6 edge-only methods that raise NotImplementedError in the base class.
ExecutionProvider
Task Runner
Dispatches async, parallel, and sub-workflow tasks. 6 required abstract methods. Controls whether work happens inline, in threads, or across Celery workers.
WorkflowObserver
Progress Reporter
Receives lifecycle events: step executed, workflow paused, compensation started, child workflow started, etc. All methods have no-op defaults — subclass only what you need.
ExpressionEvaluator
Logic Evaluator
Evaluates conditions in DECISION steps. Default implementation runs Python expressions against the current state dict.
TemplateEngine
Text Templates
Renders dynamic content — HTTP URLs, headers, bodies — from workflow state. Default implementation uses Jinja2.
WorkflowBuilder
Workflow Loader
Loads YAML definitions, resolves dotted-path imports via importlib, caches imports in a class-level dict, and wires all providers together when creating a workflow instance.

Two additional optional constructor arguments exist for advanced use cases:

wasm_runtime
WASM Runtime
Executes pre-compiled WebAssembly modules via WASI. Required only if your workflow has WASM steps.
inference_provider
ML Inference
Runs on-device model inference (TFLite, ONNX, llama.cpp). Required only if your workflow has AI_INFERENCE steps.

05 Step Types

Every step in a workflow YAML has a type field. Ten types are built in:

TypeWhat it does
STANDARDRuns a Python function synchronously. The most common type.
ASYNCDispatches a long-running task via the execution provider; workflow pauses until done.
PARALLELRuns multiple tasks at the same time; results are merged back into state.
HTTPCalls an external service over HTTP. Supports Jinja2 templating for URLs and bodies.
DECISIONEvaluates a condition and jumps to a different step via WorkflowJumpDirective.
LOOPRuns a step repeatedly over a collection or until a condition is met.
FIRE_AND_FORGETDispatches a task without waiting for the result. Good for notifications and audit logs.
CRON_SCHEDULERuns a step on a cron expression schedule (e.g. every midnight, every 5 minutes).
HUMAN_IN_LOOPPauses the workflow and waits for a human to supply input before resuming.
AI_INFERENCERuns on-device ML inference using TFLite, ONNX, or a paged LLM provider.
WASMExecutes a pre-compiled WebAssembly binary. State passed as JSON on stdin; result read from stdout.

06 The Extension System

Third-party packages can add new step types without modifying Rufus. The WorkflowBuilder uses Python's standard importlib.metadata.entry_points API to discover them automatically at startup. No config changes, no registration calls — just install the package.

How to publish an extension:

# 1. Define your step class (must subclass WorkflowStep)
class MyCustomStep(WorkflowStep):
    STEP_TYPE_NAME = "MY_CUSTOM"

    async def execute(self, state, context):
        # your logic here
        ...

# 2. Register it in your package's pyproject.toml — one line
[project.entry-points."rufus.steps"]
my_custom = "my_package.steps:MyCustomStep"

# 3. At startup, WorkflowBuilder discovers it automatically
eps = importlib.metadata.entry_points(group="rufus.steps")
for ep in eps:
    step_cls = ep.load()          # must subclass WorkflowStep
    name = getattr(step_cls, "STEP_TYPE_NAME", ep.name)
    self._step_registry[name] = step_cls

After that, any workflow YAML can use type: MY_CUSTOM and the engine will route to your class. The discovered steps participate in the full provider system — they receive the same state, context, and observer callbacks as built-in steps.

07 The Three Packages

The SDK is split into three installable packages. Start with the core; add the others only when you need them.

rufus-sdk
The engine alone (~15 MB installed)
Extras: [postgres] [cli] [performance] [otel]
rufus-sdk-edge
Adds offline storage, store-and-forward sync, config polling, and platform adapters
Extras: [edge] [browser] [wasi] [native-wasm]
rufus-sdk-server
Adds REST API, device fleet management, and command dispatch
Extras: [server] [celery] [auth] 
# Edge device only
pip install 'rufus-sdk-edge[edge]'

# Cloud server
pip install 'rufus-sdk[postgres,performance]' 'rufus-sdk-server[server,celery,auth]'

# Browser (Pyodide)
# — micropip.install('rufus-sdk-edge[browser]') from a Web Worker

08 Run Anywhere

The edge package adds a PlatformAdapter Protocol with exactly three methods. Swap the adapter and your workflow code runs on a completely different platform with no changes to step functions or YAML. 

class PlatformAdapter(Protocol):
    async def http_get(self, url: str, headers: dict) -> dict: ...
    async def http_post(self, url: str, payload: dict, headers: dict) -> dict: ...
    async def get_system_metrics(self) -> dict: ...
Native (Linux/macOS)
http_get / http_post
→ httpx async client
get_system_metrics
→ psutil
Browser (Pyodide)
http_get / http_post
→ JS fetch via FFI
get_system_metrics
→ JS performance API
WASI
http_get / http_post
→ wasi:http/outgoing
get_system_metrics
→ wasi:clocks / stub

The same YAML workflow definition runs across all three. The platform adapter is injected at startup — step functions never import platform-specific libraries directly.
