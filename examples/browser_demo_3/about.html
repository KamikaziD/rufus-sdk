1. What is this demo?

This demo runs a live peer-to-peer mesh network entirely inside browser tabs. No backend server is required. Open three tabs on the same device, or share a link to a friend on a different continent — all pods discover each other and form a self-healing gossip mesh using WebRTC DataChannels (via PeerJS) for cross-device traffic and the browser's BroadcastChannel API for same-device tabs.

Each tab is a Rufus pod running a stripped-down edge agent in a Web Worker. The pods run simulated payment transactions, gossip their capacity scores to neighbours, elect a Sovereign to coordinate relay routing, and queue offline transactions in an in-memory Store-and-Forward buffer when they simulate going offline.

Entirely serverless. No Rufus server, no database, no WebSocket relay. PeerJS uses peerjs.com only for the initial handshake (SDP + ICE). All RUVON gossip data travels directly peer-to-peer — it never touches peerjs.com.

2. Quick start — how to use it

Same-device local mesh (no setup)

  1. Serve the demo: cd examples/browser_demo_3 && python serve.py
  2. Open http://localhost:8083 in a tab.
  3. Click Local Only in the modal that appears.
  4. Open 2–4 more tabs to the same URL and click Local Only in each.
  5. Watch pods discover each other via BroadcastChannel and elect a Sovereign.

Cross-device mesh via PeerJS (no server needed)

  1. Serve the demo so other devices can reach it: 
    cd examples/browser_demo_3 && python serve.py
# prints: Other devices: http://192.168.x.x:8083
  2. Device A: open the URL → enter a group key (e.g. team-42) → Join Mesh ↗
  3. Device B: open the same URL → enter the same group key → Join Mesh ↗
  4. Or: Device A clicks Share Link in the header — paste the URL on Device B.
  5. Watch "PeerJS connected" and "Joined existing mesh room" appear in both event logs.
Works anywhere you can serve static files — GitHub Pages, VS Code port forwarding, file:// (same device only). No port 8000 required.

Simulating offline / SAF

  1. In any pod, click Go Offline. The pod badge turns red.
  2. Trigger a transaction — it queues as SAF (yellow warning in the log).
  3. Click Go Online. The SAF queue drains immediately.

3. RUVON: the gossip protocol

RUVON (Rufus Vectored Observation Network) is a lightweight gossip protocol running entirely in a Web Worker. Each pod periodically broadcasts a signed heartbeat to all neighbours. A heartbeat carries:

FieldTypeDescription
pod_idstringUnique ID generated at pod startup (8-char hex)
score0–1 floatCapacity score S(Vc) — see §4
tier1 / 2 / 3Capacity tier derived from score
vecobjectRaw vector: cpu, heap, uptime, pending_count
is_sovereignboolTrue if this pod currently holds the Sovereign role
saf_depthintNumber of queued SAF transactions
propagated_countintNumber of re-broadcast hops (damping)
timestampms epochUsed for dedup and stale-peer eviction

Heartbeats propagate through the mesh in a fan-out with damping: each pod re-broadcasts a received heartbeat once — but only if propagated_count < 15 and the message was not seen before (LRU dedup cache, 50 entries). This prevents broadcast storms while ensuring every pod hears about every other pod within 2–3 hops.

Message types

TypeDirectionPurpose
HEARTBEATbroadcastPeriodic capacity advertisement
ELECTIONbroadcastPropose a new Sovereign (highest scorer)
RELAY_REQUESTunicast → SovereignOffline pod delegates a transaction
RELAY_RESULTunicast → requesterSovereign returns execution result
HELP_REQUESTmulticast → top-3Distressed pod asks for support
HELP_OFFERunicast → requesterHealthy pod acknowledges help request

4. Capacity scoring S(Vc)

Every pod computes a scalar score from its resource vector:

S(Vc) = 0.50·C + 0.15·(1/H) + 0.25·U + 0.10·P
ComponentWeightSourceMeaning
C — CPU slack0.501 − cpu_usageAvailable CPU headroom (dominant factor)
H — Heap pressure0.151 / heap_ratioInverse of memory used / total
U — Uptime bonus0.25min(uptime_s / 120, 1)Stability — new pods score lower
P — Queue penalty0.101 − min(saf_queue/10, 1)SAF backlog reduces capacity

In the browser, cpu_usage and heap_ratio come from performance.memory (Chromium) with random jitter to simulate real device variation. The score updates on every heartbeat cycle.

5. Adaptive gossip intervals

Unlike a fixed-interval heartbeat, RUVON adjusts how often each pod gossips based on its own score — healthy pods gossip fast, stressed pods gossip slow: 

// Base interval: 2000 ms
interval = BASE_INTERVAL / score^1.2
           + jitter(±25–50%)        // prevents synchronised spikes

// Examples:
// score=0.95 (T1 healthy)  → ~300 ms gossip
// score=0.50 (T2 average)  → ~1.8 s gossip
// score=0.10 (T3 distress) → ~10 s gossip

Sovereign Pulse Mode: the elected Sovereign always uses exactly BASE_INTERVAL_MS (2 s) regardless of score — it needs to be reliably reachable for relay requests.

Help-push trigger: if a pod's score drops by more than 0.15 in a single cycle, or its SAF queue reaches ≥ 3, it immediately sends a HELP_REQUEST to the 3 highest-scoring peers. Those peers reply with a HELP_OFFER — logged in the event panel.

6. Tier system & Sovereign election

TierScore rangeRoleRe-broadcasts?
T1 ≥ 0.70 High-capacity — eligible for Sovereign Only if Sovereign
T2 0.40–0.70 Standard relay pod Yes
T3 < 0.40 Degraded — consumes only Yes (reduced fanout)

Sovereign election

When a pod sees its own score is higher than the current Sovereign's last-known score, it broadcasts an ELECTION message nominating itself. All pods update their view of the Sovereign to whichever pod has the highest score. The election is eventually consistent — brief split-brain is harmless because relay requests are small and idempotent.

Stale peers (no heartbeat for > 8 s) are automatically evicted from the peer table. If the Sovereign goes stale, the next heartbeat from any T1 pod triggers a new election.

7. Transport layer — BroadcastChannel + PeerJS

The demo uses two complementary transports:

TransportScopeAPILabel
BC BroadcastChannel Same device (all tabs, same origin) Web standard, zero setup Green BC in pod table
P2P PeerJS DataChannels Cross-device (any network) WebRTC via peerjs.com signaling Blue P2P in pod table

The worker only knows about broadcastAll(msg) — it posts to BroadcastChannel and sends {type:"REMOTE_SEND"} to the main thread. The main thread fans the message out to all open PeerJS DataChannels. Incoming DataChannel messages arrive as {type:"REMOTE_MSG"} and are forwarded to the worker for dedup + processing.

This split keeps WebRTC (which needs window) in the main thread and all RUVON logic in the isolated worker — a clean architecture boundary.

8. How PeerJS mesh discovery works

PeerJS requires each peer to know the remote peer's ID before connecting. The demo solves discovery without any server using the anchor pod pattern:

  1. Every pod registers with PeerJS using the ID rufus-{groupKey}-{podId}.
  2. Simultaneously, every pod tries to connect to the well-known anchor rufus-{groupKey}-anchor.
  3. If the anchor responds (room already exists): the anchor sends a _MEMBER_LIST message → the new pod connects directly to all listed peers.
  4. If no anchor responds within 2.5 s: this is the first pod in the group. It registers a second PeerJS peer with the anchor ID and starts relaying member lists to newcomers.
  5. The anchor is just a regular pod wearing two hats. If it disconnects, the next pod to join will time out on the anchor and claim the anchor slot — fully self-healing.
┌─────────────────────────────────────────────────────┐ │ peerjs.com (signaling only — never sees payload) │ │ SDP/ICE exchange: Pod A ↔ Pod B │ └──────────────────────┬──────────────────────────────┘ │ handshake only ┌───────────┴────────────┐ │ │ ┌────▼────┐ ┌────▼────┐ │ Pod A │◄────────────►│ Pod B │ │ (anchor)│ WebRTC DC │ │ └────┬────┘ └────┬────┘ │ BroadcastChannel │ ┌────▼────┐ ┌────▼────┐ │ Tab A2 │ │ Pod C │ │(same dev│ │(device3)│ └─────────┘ └─────────┘
peerjs.com privacy: The signaling server only sees peer IDs and SDP handshake data. It never receives or forwards RUVON heartbeats, transaction data, or any group key payload. All mesh traffic flows directly device-to-device.
Note: PeerJS requires internet access to reach peerjs.com for the initial handshake. Once connected, peers communicate directly via WebRTC — the signaling server is no longer involved. If peerjs.com is unreachable, the demo falls back to BroadcastChannel-only (same-device tabs still work).

9. Store-and-Forward (SAF)

SAF simulates one of Rufus's core fintech features: processing payments when the network is unavailable, then syncing when back online.

  1. Pod detects offline state → queues transaction in-memory SAF buffer.
  2. SAF depth reflected in score vector (increases queue penalty P).
  3. On reconnect → all queued transactions drain in order.
  4. Alternatively: if the pod has a Sovereign neighbour, it sends a RELAY_REQUEST so the Sovereign can execute the transaction on its behalf while the local pod is offline.

In the browser demo, "online" and "offline" are simulated states — no actual network change occurs. In production Rufus deployments, the SAF layer uses SQLite for durability.

10. Architecture diagram

index.html (main thread) ├── Modal UI + share link ├── BroadcastChannel("rufus-mesh") ←── same-device tabs ├── PeerJS DataChannels ←── cross-device peers │ └── relay REMOTE_SEND ↔ REMOTE_MSG with worker └── Canvas / peer table / event log renderer worker.js (Web Worker) ├── RUVON engine: heartbeat scheduler, peer table, election ├── Workflow runner: SAF queue, relay logic ├── broadcastAll(msg) │ ├── BroadcastChannel.postMessage(msg) → local tabs │ └── self.postMessage({type:"REMOTE_SEND", msg}) → main thread → PeerJS └── self.onmessage ├── INIT / STOP / TOGGLE_OFFLINE / RUN_TX └── REMOTE_MSG ← arrives from main thread via PeerJS DataChannel

The worker never touches the DOM or WebRTC APIs — it is a pure logic layer that posts structured messages to the main thread for all I/O.

11. Test scenarios

Scenario A — Tier cascade

Open 5 tabs. After ~30 s the tier badges stabilise. Notice how the fastest-gossiping pod (shortest BC interval) tends to win Sovereign. Open DevTools Performance tab on a high-load tab and reduce its CPU budget — watch it drop to T3 and lose Sovereign.

Scenario B — SAF relay

Open 2 tabs. Click Go Offline on tab 1 so it has a Sovereign (tab 2). Click Run Transaction on tab 1 — observe the RELAY_REQUEST logged in both tabs. Tab 2 executes the transaction and returns the result. Click Go Online to drain any queued SAF entries.

Scenario C — Cross-device latency

Join from two devices on the same Wi-Fi. Open the event log on both. Run a transaction on the device with the lower score — it relays to the higher-score Sovereign on the other device. The round-trip latency appears as the gap between RELAY_SENT and RELAY_RECEIVED log entries.

Scenario D — Anchor self-healing

Join 3 devices. Close the tab that became the anchor (the first one to join). Wait ~10 s for stale-peer eviction. Open a new fourth tab with the same group key. It will time out on the missing anchor and claim the anchor role itself — the mesh re-forms without any manual intervention.

Scenario E — GitHub Pages (no server at all)

Push examples/browser_demo_3/ to a gh-pages branch. Open the GitHub Pages URL on two different devices. Enter the same group key. The mesh forms via peerjs.com — zero infrastructure required.
